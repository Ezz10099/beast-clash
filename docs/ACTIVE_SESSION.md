## Current Work Packet

- **Status:** In progress.
- **Player-facing problem:** An Arabic-speaking fresh player may fail to understand essential instructions and spell choices because of English vocabulary, making the commercial cell measure language proficiency instead of game comprehension, enjoyment, pacing, and replay desire.
- **Intended player experience:** The player should independently understand movement, hazards, automatic casting, the three spell axes, rewrite consequences, progress, and replay selection while experiencing the exact accepted `.8` gameplay.
- **Strongest known cause:** Essential meanings are encoded in English DOM and canvas text; visual cues help but do not fully explain Form, Essence, Law, rewrite outcomes, results, and Spellbook progression.
- **Alternatives considered:**
  1. Verbally translate during the test — rejected because coaching contaminates the non-leading cell and is inconsistent.
  2. Fully localize the release now — rejected as premature scope expansion before the commercial gate.
  3. Add a bounded query-activated Arabic test mode with RTL, essential DOM/canvas localization, no gameplay/save changes, and automated preservation checks — selected.
  4. Test in English despite the constraint — rejected because failure would be uninterpretable.
- **Selected approach:** Add a small standalone localization layer loaded before `game.js`, enabled only by `?lang=ar`. It will translate essential static and dynamic DOM text plus canvas labels, set Arabic/RTL presentation, preserve English as default, and coexist with `?fresh=<token>` save isolation.
- **Predicted visible benefit:** A fresh Arabic-speaking player can understand what to do and what choices mean without human coaching, allowing the cell to evaluate the game rather than English vocabulary.
- **Likely failure mode:** Partial translation, awkward Arabic, RTL overflow, or automatic string translation missing a dynamic state could introduce confusion or alter visual readability.
- **Evidence required:** Deterministic checks for default-English preservation, `?lang=ar` activation, RTL/language state, essential Arabic strings, fresh-save isolation compatibility, and unchanged gameplay evidence; one consolidated owner SPCK check for Arabic meaning and phone layout before the fresh-player cell.
- **Files/systems expected to change:** `index.html`, `style.css`, a standalone localization script, deterministic checks, package version, and session/handoff documentation. `game.js` gameplay logic should remain unchanged unless a proven localization gap cannot be solved safely outside it.
- **Approval boundary:** Test-readiness localization only. No gameplay, balance, progression, persistence, content scaling, final art, monetization, identity, or release action changes.
